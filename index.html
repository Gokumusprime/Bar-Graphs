Below is a graph that was constructed using the chart.js library. I decided to use this because it would allow for us to make these graphs reusable in a CMS environment through HTML markup. I’ll go over that part more in detail down the page.

Here’s what I’ve added to the graph to improve it:

  • The graph is no longer an image.
  • The graph is now responsive while maintaining a good look.
  • The graph now uses the canvas HTML5 element which is in line with modern browsers.
  • The graph now animates on load and when scrolled to.
  • I’ve added a hover effect when hovering over a bar in the graph.
  • I included a tooltip on any particular bar once hover over it. The tooltip shows the exact value for the current bar.
  • It functions in all modern browsers.

Now, what do I mean by this graph being easily re-suable and editable? What I’ve done here is create js that works with the chart.js library to detect hidden element content on the page. Of course, we could later set this content as variables but I wanted this to function easily in side of a CMS at the moment.

When trying to create a horizontal bar graph, a user could edit the RTE field inside Wordpress or any other CMS and just drop some simple html markup. The span tags inside this markup would contain your raw delimited data for the graph. See the snippet markup in the image below.

A single page is also capable of having multiple graphs in this manner. I would later investigate different types of snippet graphs or find a way to utilize a field in the CMS that would populate a json file or js variables to use instead.

Question 2: What Would I Change About Klaviyo.com

  • Located on this page: https://www.klaviyo.com/marketing-resources/ecommerce-benchmarks , I would combine the “Avg Revenue Per Recipient By Email Type” graphs into a single graph with filter buttons to adjust the average order size. That way you conserve space and allow the user to switch between average. It may even be possible to add an input to allow the user to put in their own average to see how it shapes out.
  • The ability to share content on social media for pages like https://www.klaviyo.com/marketing-resources/ecommerce-benchmarks only appears when scrolling past a certain point. I understand that it’s because this is using the custom header that contains page anchors and you may not want to display it when scrolled to the top. In this case, I would have the social media share icons either statically placed on the side of the screen above the content so that it’s always there while scrolling or I would only place it there once you begin the scrolling and the header appears. You could reuse the same html markup for the social media to move to a fixed look on scroll by adding a class to it.
  • I found the pop-up modal a bit intrusive on the marketing resource pages. For example: https://www.klaviyo.com/marketing-resources/ecommerce-maturity-model . It appears right in front of you while scrolling. I understand that we wouldn’t want clients or potential clients to miss it so I would maybe have it animate out from a fixed position of the left side of the screen when scrolling past the same point. It will only occupy the available white space on the side of the screen and won’t block what you’re reading and forces you to take notice at the same time. For a mobile solution, I would have it fixed to the bottom of the screen in front of the content by altering its z-index. While at the bottom, it would look like a thinner strip of content that spans across the bottom part of the page. It will feel like it’s a part of the browser’s UI on a mobile phone.

Question 3: How I Would Create Chart 2

I would use the following libraries:

  • Bootstrap.css and js
  • D3.js or chart.js (Though chart.js is nice, d3.js may be better for more complicated goals)
  • Possibly angular js for modularizing graphs into re-usable components

I would use the bootstrap simply to keep the surrounding div responsive around the graph at different device breakpoints. This could be done by using specific bootstrap classes in the markup. I would use the SASS include-media mixin library to make any alterations to the graph itself at different breakpoints. The js libraries for charts will be taking care of most of the responsive work but it’s good to have some backup styling for what is missing.

To build out the javascript, I would focus on a json source possibly from a web service or one planted on the page by entering raw data into the CMS. The json would be broken up into different indexes for the different filtered options. From there, each index would contain the raw data needed for that view of the graph.

JavaScript would then be constructed for building the graph using the horizontal bar graph option that would target simple markup on the page. A function would execute this graph’s construction on load but would also allow you to execute it once more when filter buttons are clicked. When it executes, it will consume the json, construct the graph and append it to the HTML markup on the page. If the filter buttons are clicked, the function would be passed a variable containing the parsed out json that’s needed for that filter. This in turn will re-animate the graph with altered raw data while keeping it in place due to the fancy chart js libraries that I would be using.

Now I come from a CMS world and I strongly believe that constructing features like this needs to be re-usable. I know that I talked about this with the first question but I would eventually steer down a path of using angular.js to make these graphs components that could be dropped anywhere through a CMS.
